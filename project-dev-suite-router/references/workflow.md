# 项目开发全流程 Skill

按照软件工程瀑布流规范，从想法到交付的标准化开发流程。

## 触发场景

- 用户说"我要做一个新项目"
- 用户说"帮我从头开始规划"
- 用户有一个想法想要系统化落地

## 流程总览（瀑布流）

Phase 1: 需求分析
    ↓
Phase 2: 任务拆解（子需求 + 开发任务 + 里程碑）
    ↓
Phase 3: 原型设计 + 流程图（可并行）
    ↓
Phase 4: 架构设计（智能推荐技术栈）
    ↓
Phase 5: 数据库设计
    ↓
Phase 6: 需求转化（售前→产品）
    ↓
Phase 7: 详细需求 + 接口文档
    ↓
Phase 8: 代码生成
    ↓
Phase 8.5: 冒烟测试（骨架能跑通）
    ↓
Phase 9: 测试生成 + 执行
    ↓
Phase 10: 迭代开发（按任务清单逐个交付）
```

## Phase 1: 需求分析

**Skill**: `requirements-analysis`

**输入**: 用户的粗略想法
**输出**: `docs/requirements/[售前需求]{项目名}.md`

**检查点**: 核心对象、功能菜单、业务流程、数据模型 → 用户确认

## Phase 2: 任务拆解

**Skill**: `task-breakdown`

**输入**: 已确认的售前需求文档
**输出**: `docs/tasks/开发任务清单.md`

**执行步骤**:
1. 从需求文档提取功能模块 → 形成子需求清单（SR-001/002/...）
2. 每个子需求拆解为开发任务（TASK-xxx-xx，含类型/工时/依赖）
3. 生成任务依赖图（Mermaid 甘特图）
4. 划分里程碑（M1/M2/M3/...）
5. 确定推荐开发顺序

**检查点**: 子需求划分合理、任务粒度适中（1-8h）、依赖无循环 → 用户确认

**后续阶段按里程碑推进**: Phase 3-10 按 M1→M2→M3 的顺序逐个里程碑执行

## Phase 3: 原型设计 + 流程图

**Skill**: `prototype-generator` + `flowchart-generator`
**可选MCP**: `chrome-devtools`（预览原型）

**输入**: 售前需求 + 任务清单（按当前里程碑范围生成）
**输出**: HTML原型 + Mermaid流程图 + HTML关联图

## Phase 4: 架构设计

**Skill**: `architecture-design`

**输入**: 售前需求 + 原型 + 任务清单
**输出**: 架构文档 + 架构图HTML

**关键动作**:
1. 分析项目特征 → 智能推荐技术栈 + 理由
2. 用户确认或调整（给出风险提示）
3. 锁定技术栈，后续所有阶段按此执行

## Phase 5: 数据库设计

**Skill**: `database-design`
**MCP**: `mysql-custom`（执行建表+验证）

**输入**: 架构文档 + 需求中的数据模型
**输出**: `database/V1__init_tables.sql`

## Phase 6: 需求转化

**Skill**: `requirements-transform`

**输入**: 售前需求 + 原型
**输出**: `[需求文件]xxx.md` + `需求原型-xxx.html`

## Phase 7: 详细需求 + 接口文档

**Skill**: `detail-requirements` + `api-doc-generator`

**输入**: 产品需求 + 架构 + 数据库设计 + 任务清单
**输出**: 细化的需求文件 + 接口文档

**按子需求逐个细化**: 先细化当前里程碑涉及的子需求

## Phase 8: 代码生成

**Skill**: `code-scaffold`

**输入**: 架构文档 + 数据库设计
**输出**: 完整可运行的项目骨架代码

## Phase 8.5: 冒烟测试

**在代码生成后、正式测试前，快速验证骨架代码能正常运行。**

**目的**: 尽早发现配置级问题（端口冲突、依赖缺失、数据库连不通），避免等到 Phase 9 才暴露

**冒烟测试清单**:
```
1. 项目能编译/启动，无报错
2. 健康检查接口可访问（如 GET /health 返回 200）
3. 数据库连接正常（表已创建、可读写）
4. 基础 CRUD 接口能响应（至少一个模块的增删改查）
5. 认证流程可走通（如有，登录拿到 token）
```

**执行方式**:
- 启动服务 → 逐项手动或脚本验证
- 配合 MCP: mysql-custom 验证数据库、chrome-devtools 验证前端页面

**结果判定**:
- 全部通过 → 进入 Phase 9 正式测试
- 任一失败 → 修复后重新冒烟，不进入 Phase 9

## Phase 9: 测试生成 + 执行

**Skill**: `test-generation`
**MCP**: `mysql-custom` + `chrome-devtools`

**输入**: 需求文档 + 代码
**输出**: 测试用例 + 测试报告

**测试通过后才进入 Phase 10**

## Phase 10: 迭代开发

**Skill**: `lint` + `find-bugs` + `test-generation`
**MCP**: `mysql-custom` + `chrome-devtools`

**按任务清单逐个执行**:
```
取任务清单中下一个待开发任务
    ↓
实现功能代码
    ↓
lint 检查
    ↓
test-generation 生成并执行测试
    ↓
    ├── 通过 → 标记任务完成 → 取下一个任务
    └── 失败 → 修复 → 重新测试
    
每完成一个里程碑 → find-bugs 安全审查
```

### 任务收尾流程（每个任务完成后必须执行）

每个任务测试通过后，按以下步骤收尾：

1. **更新任务状态**：在 `docs/tasks/开发任务清单.md` 中将当前任务状态改为「已完成」
2. **Git 提交**：提交代码，commit 信息格式：
   - 功能类：`feat(SR-xxx): TASK-xxx-xx 任务描述`
   - 修复类：`fix(SR-xxx): TASK-xxx-xx 修复描述`
   - 数据库变更类：`db(SR-xxx): TASK-xxx-xx 表结构描述`
3. **经验回写**：触发 AGENTS.md 中的经验回写流程（第三步），判断本次任务是否产生了可沉淀的经验
   - 通用经验 → 写入 `D:\AI\knowledge-base\` 对应子目录
   - 项目专属经验 → 写入项目 `docs/` 对应子目录

### 里程碑收尾（一个里程碑的所有任务完成后）

1. **安全审查**：执行 `find-bugs` 对本里程碑的所有变更做安全审查
2. **里程碑标记**：在任务清单中标记里程碑为「已完成」
3. **Git Tag**：打 tag 标记里程碑，格式：`M1-基础可运行`、`M2-核心流程跑通`

## MCP 集成说明

| MCP | 使用阶段 | 作用 |
|---|---|---|
| mysql-custom | Phase 5, 9, 10 | 建表、测试数据、数据验证 |
| chrome-devtools | Phase 3, 9, 10 | 预览原型、E2E测试、前端调试 |

## 流程控制规则

1. **每阶段有卡点**: 用户确认后才进入下一阶段
2. **可回退**: 任何阶段发现问题可回到上一阶段
3. **可并行**: Phase 3（原型+流程图）可并行
4. **里程碑驱动**: Phase 7/9/10 按里程碑增量执行
5. **MCP可选**: 没有MCP也能完成全流程
6. **技术栈不固化**: Phase 4 智能推荐，用户有最终决定权
7. **测试驱动**: Phase 9 测试必须通过才能进入 Phase 10
8. **任务驱动**: Phase 10 按任务清单逐个交付，每个任务有明确验收标准

## 进度追踪

每次对话开始时检查项目阶段：
1. 有 `[售前需求]` → Phase 1 完成
2. 有 `docs/tasks/` → Phase 2 完成
3. 有 `docs/prototypes/` 主架构页 → Phase 3 完成
4. 有 `docs/architecture/` → Phase 4 完成
5. 有 `database/` SQL文件 → Phase 5 完成
6. 有 `[需求文件]` → Phase 6 完成
7. 有 `docs/api/` → Phase 7 完成
8. 有 `src/` 代码 → Phase 8 完成
9. 服务可启动 + 基础接口可访问 → Phase 8.5 冒烟通过
10. 有 `tests/` 且测试通过 → Phase 9 完成
11. 按任务清单检查完成度 → Phase 10 进行中


## V1.1 流程覆盖规则

以下规则优先于前文中与之冲突的阶段顺序或文档约定。

### 核心文档契约

项目启动后，以以下文件作为事实源：

```text
docs/PRD.md             产品定位、MVP、非目标、范围与验收
docs/DESIGN.md          有 UI 时的视觉、组件与交互状态约束
docs/ARCHITECTURE.md    技术栈、目录边界、运行约束与不可破坏规则
docs/TODO.md            当前可执行任务及其验收和测试要求
```

详细材料仍放在 `docs/requirements/`、`docs/prototypes/`、`docs/architecture/`、`docs/api/`、`docs/tasks/` 与 `database/`。核心文档冲突时先向用户确认，不得自行选择。

### 更新后的阶段顺序

1. **产品定位与可行性预检**：澄清用户、痛点、替代方案、差异化、MVP 与非目标、产品形态、成本、约束和风险。
2. **PRD 确认**：生成 `docs/PRD.md`，用户确认范围后才进入设计或架构。
3. **设计契约与原型验证（有 UI 时）**：先生成 `docs/DESIGN.md`，再制作原型和流程图；原型发现问题时回写 PRD。
4. **架构与技术可行性确认**：生成 `docs/ARCHITECTURE.md`，确认技术栈、模块边界、AI/外部服务、数据、安全、部署和成本风险。
5. **两段式任务计划**：PRD 后先生成里程碑路线图；PRD、DESIGN（如适用）和 ARCHITECTURE 确认后，再生成细粒度 `docs/TODO.md`。
6. **详细需求、接口与数据库设计**：按当前里程碑落地，不得突破核心文档边界。
7. **代码骨架与冒烟测试**：编码前读取相关核心文档；冒烟失败不得进入正式迭代。
8. **小步迭代**：一次只做一个 TODO，完成后验证主流程、加载、空、错误和失败路径，再提交。
9. **发布就绪与文档反扫**：使用 `references/templates/RELEASE-READINESS.md`，以最终代码回写架构、环境变量、部署说明与 README。

### 两段式任务规则

- 路线图只描述里程碑、范围、依赖和风险。
- 可执行 TODO 必须包含：目标、允许修改范围、不可破坏项、验收标准、测试要求、依赖和状态。
- 单个开发任务应在 1-8 小时内完成；超出时继续拆分。
- 小型 POC 可以合并阶段，但不得跳过产品边界、架构约束和基础验证。


## V1.2 持续版本迭代与维护治理

发布不是流程终点。每次首发、版本发布或热修复后，都回到路线图与项目事实源，开始下一轮治理。

### 路径一：新项目

新项目继续按 V1.1 的产品定位、PRD、设计、架构、两段式计划、开发、发布路径推进。MVP 是首个受控版本，不代表产品生命周期结束。

### 路径二：已有项目的版本迭代

适用于 V1.1、V1.2、V2.0 等功能版本，不重新进行完整的新项目定位。

1. 读取当前 `PRD.md`、`DESIGN.md`（如存在）、`ARCHITECTURE.md`、`ROADMAP.md`、`TODO.md`、最近发布记录和相关代码。
2. 使用 `references/templates/VERSION-PLAN.md` 创建 `docs/releases/{版本号}/VERSION-PLAN.md`。
3. 明确本版本目标、范围、非目标、受影响模块、数据/API/权限/成本风险和回归范围。
4. 仅在必要时更新 PRD、DESIGN、ARCHITECTURE；所有变更记录必须可追溯。
5. 生成本版本 TODO，按小步迭代实现。
6. 执行目标测试和受影响范围的回归测试。
7. 使用发布检查清单，更新 README、架构、部署说明、已知限制和 ROADMAP。
8. 创建版本 Git tag，并将本版本结果沉淀到发布记录。

### 路径三：维护、缺陷与紧急修复

适用于 bug、性能退化、安全问题、线上故障和依赖风险。

1. 先确认严重等级、影响用户、复现条件和受影响版本。
2. 使用 `references/templates/MAINTENANCE-RECORD.md` 记录问题、最小修复范围、风险和回归范围。
3. 不借修复机会混入无关重构或新功能；需要扩大范围时先获得用户确认。
4. 完成定向测试、受影响主流程回归和安全/质量检查。
5. 修复发布后更新维护记录、相关事实源、README 或运行手册，并创建 hotfix Git tag。
6. 对重复性问题补充根因、预防措施和后续 TODO。

### 文档生命周期

- `PRD.md` 保存产品愿景和所有版本范围变化。
- `DESIGN.md` 随组件和交互规则演进。
- `ARCHITECTURE.md` 随真实技术结构、依赖和部署方式演进。
- `ROADMAP.md` 保存跨版本路线图。
- `TODO.md` 只代表当前版本或当前迭代，不应成为永久待办池。
- `docs/releases/` 保存各版本计划与发布记录。
- `docs/maintenance/` 保存维护和事件记录。

### 持续循环

```text
发布或修复完成
-> 更新事实源和路线图
-> 收集反馈、数据、风险或新需求
-> 选择版本迭代或维护路径
-> 影响分析
-> TODO、开发、回归、发布
-> 进入下一轮
```

## V1.3 发布前风险驱动测试与安全攻防门禁

本规则只在本次版本范围内的功能开发完成后、版本发布前、重大功能集合合并前或高风险 hotfix 发布前执行。完整门禁不针对每个单独 TODO 重复执行。它不是需求阶段，也不是架构阶段；PRD、DESIGN、ARCHITECTURE、TODO、VERSION-PLAN 和 MAINTENANCE-RECORD 只作为测试输入。

### 执行位置

```text
本次版本/重大功能集合开发完成
-> 常规单元/集成/E2E 测试
-> V1.3 风险驱动功能测试与安全攻防门禁
-> 问题清单与方案核对
-> 修复与回归
-> 发布就绪检查
-> 发布 / Git tag
```

### 独立测试产物

发布前必须在项目中生成或更新以下独立产物，不要塞进 PRD 或 ARCHITECTURE：

```text
docs/tests/RISK-BASED-TEST-PLAN.md
docs/tests/TEST-AUTOMATION-RUNBOOK.md
docs/tests/TEST-EFFECTIVENESS-REPORT.md
docs/tests/SECURITY-ATTACK-REPORT.md
```

### 门禁步骤

1. **高风险功能识别**：按资金、权限、数据三类风险列出必须重点测试的前后端功能，同时列出暂不重点测试的低风险功能，并说明原因。
2. **关键功能测试设计**：只对高风险功能深测，每个关键功能覆盖正常正确、边界临界、异常非法三类情况；重点验证业务结果和数据变化，而不是只看接口是否返回成功。
3. **自动化测试命令固化**：将已确认的关键测试封装成一条或少数几条命令，记录怎么运行、怎么看通过/失败、单个任务如何局部复跑、版本发布前如何完整复跑。
4. **测试真实性验证**：逐个说明测试验证了什么业务结果；对关键测试通过受控故障注入验证“代码改错时测试必须失败”。若代码错了测试仍通过，该测试视为假测试，必须重写。
5. **安全攻防测试**：在授权范围内，以安全专家/攻击者视角审查后端代码并实测接口，覆盖数据越权、功能越权、身份认证、字段提权、恶意输入与注入、绕过前端限制、敏感信息泄露、资源滥用与防刷。
6. **问题清单与方案核对**：发现问题后先列风险、后果、复现证据和建议方案，不直接改代码；逐条与用户确认真实存在、方案合理且不会破坏其他功能。
7. **修复与回归**：用户确认后逐条修复，全部修完后重跑常规测试、风险测试、安全攻防验证和发布检查。

### 深度原则

- 资金、权限、数据相关功能深测、深防、深回归。
- 次要功能验证基本可用即可，避免为了覆盖率写重复或低价值测试。
- 若无法执行某项测试，必须记录原因、风险和替代验证方式。
## V1.3.1 上游轻量风险标记

V1.3 的测试主体仍然只在发布前门禁集中执行，但上游文档需要留下轻量风险输入，方便测试阶段快速锁定重点。

- PRD：功能清单应标记是否涉及资金、权限、数据，以及出错后果；不展开完整测试用例。
- ARCHITECTURE：说明权限、数据、接口、安全边界和不可破坏规则；不执行攻防测试。
- TODO / VERSION-PLAN / MAINTENANCE-RECORD：标明任务、版本变更或修复是否涉及资金、权限、数据，以及对应回归范围；不写大量测试脚本。
- 发布前测试门禁：集中生成测试计划、自动化运行手册、测试有效性报告和安全攻防报告，并完成用户核对后再修复。
## V1.4 轻量推理原则

本版本不新增阶段、不新增产物，只为关键决策补充两条思考原则。

### 第一性原理使用边界

第一性原理不是拒绝成熟方案，也不是从零发明一切。仅在高影响决策中使用：产品定位、MVP 边界、技术选型、权限模型、资金规则、数据一致性、AI 成本/隐私、多租户、安全边界和发布回滚。

执行方式：先拆目标、约束和不可破坏规则，再参考成熟方案，最后判断哪些可复用、哪些不适合照搬。普通 CRUD、简单页面和低风险功能优先使用成熟实践，不做过度推导。

### 对抗式审查使用边界

对抗式审查不是新增独立流程，而是关键节点的检查姿态。用于 PRD 确认、架构确认、TODO 执行前、版本发布前、维护/热修复前后，重点追问：方案会在哪里失败、如何被误用或滥用、是否破坏资金/权限/数据/可靠性/发布安全。

已有 V1.3 风险测试与安全攻防门禁仍是主要落点，不额外要求每个阶段生成对抗式审查报告。
### 测试门禁触发粒度

- 单个 TODO 完成后：只执行该任务相关的单元测试、接口测试、lint/typecheck，以及必要的局部风险验证。
- 单个高风险功能完成后：如果涉及资金、权限或数据，提前执行该功能的局部风险测试和必要安全验证。
- 版本发布或重大功能集合合并前：执行完整 V1.3 风险驱动功能测试与安全攻防门禁，生成并核对四个独立测试产物。
- 高风险 hotfix 发布前：按受影响范围执行精简但严格的风险门禁、定向测试和回归测试。

## V1.5 流程强度分级与上下文节省

V1.5 不新增开发阶段，新增的是“调度规则”：先判断这次任务需要多重的流程，再决定读取多少上下文、生成哪些产物、执行多深的验证。

### 强度分级

- **L0：不启用项目套件**。适用于普通问答、代码解释、单文件小修、一次性脚本、文档/图片/表格等非软件项目任务。保持普通 Codex 行为，不读取项目流程引用。
- **L1：轻量上下文模式**。适用于已有项目里的小功能、小 bug、小优化。只读取当前相关代码、当前 TODO 或必要事实源片段；只做局部测试和必要局部风险验证。
- **L2：标准项目迭代模式**。适用于一个明确版本或一组有业务意义的功能。读取 PRD、DESIGN（如涉及 UI）、ARCHITECTURE、TODO、版本计划和相关代码；更新 VERSION-PLAN/TODO；执行范围内测试和回归。
- **L3：发布或高风险门禁模式**。适用于版本发布前、重大功能集合合并前、高风险 hotfix 发布前，或涉及资金、权限、核心数据、安全边界的高风险变更。执行 V1.3 风险驱动测试与安全攻防门禁。

### 项目状态识别输出

启用套件后，先用简短状态扫描对齐，不要直接进入完整流程：

```text
当前识别：
- 路径：新项目 / 版本迭代 / 维护热修复
- 强度：L1 / L2 / L3
- 当前阶段：需求 / 设计 / 架构 / 开发 / 测试 / 发布前 / 维护
- 已有事实源：PRD / DESIGN / ARCHITECTURE / TODO / VERSION-PLAN / MAINTENANCE-RECORD / 测试报告
- 缺失或冲突事实：
- 高风险区域：资金 / 权限 / 数据 / 安全 / AI 成本隐私 / 无
- 下一步：
```

### 上下文加载预算

默认只加载当前任务必要上下文；发现冲突、缺口或高风险时再扩大读取范围。

- L1：读取当前任务、相关代码、必要 ARCHITECTURE/TODO 片段；UI 变更才读取 DESIGN；产品范围不清才读取 PRD。
- L2：读取 PRD、ARCHITECTURE、TODO、VERSION-PLAN 和相关代码；涉及 UI 才读取 DESIGN；涉及数据/API/权限才读取对应详细文档。
- L3：在 L2 基础上读取发布检查和四个测试产物模板；只对本次版本或受影响范围执行完整门禁。

### 影响范围与完成定义

版本迭代和 hotfix 必须明确直接影响、间接影响、数据影响、API 影响、权限影响、UI 影响、回归范围和不应修改范围。

单个 TODO 完成定义：功能实现、相关测试通过、未突破允许修改范围、未破坏不可破坏项、TODO 状态更新，必要时完成 Git 提交。

版本完成定义：本版本 TODO 完成、常规测试通过、需要时完成 L3 风险门禁、文档反扫完成、发布记录和 Git tag 完成。

Hotfix 完成定义：问题可复现、最小修复完成、定向测试通过、受影响路径回归、维护记录更新、必要时完成 hotfix tag。
