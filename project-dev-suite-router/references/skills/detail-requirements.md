# 详细需求细化 Skill

将售前需求文档转化为开发可直接执行的详细需求文件。

## 触发场景

- 用户说"帮我细化需求"、"写详细需求"
- 售前需求确认后进入详细设计阶段
- 用户要求把某个功能模块拆解到字段级别

## 前置条件

- 项目中已有售前需求文档
- 如果没有，先引导用户使用 requirements-analysis skill

## 执行流程

### Phase 1: 选择细化范围

1. 读取项目中的售前需求文档
2. 列出所有功能模块，让用户选择要细化的模块
3. 一次只细化一个模块，避免文档过长

### Phase 2: 生成详细需求文档

```markdown
# [需求文件]{功能模块名}

## 功能概述
一句话描述本模块的核心能力。

## 角色与权限
| 角色 | 可执行操作 | 数据范围 |
|---|---|---|
| 管理员 | 全部 | 全部 |
| 普通用户 | 查看/新增 | 本人数据 |

## 功能清单
### 1. {功能点名称}
**触发条件**：用户点击xxx / 系统定时触发
**前置条件**：用户已登录 + 有xxx权限
**主流程**：
1. 步骤1
2. 步骤2
3. ...

**字段定义**：
| 字段 | 类型 | 必填 | 规则 | 说明 |
|---|---|---|---|---|
| 名称 | 文本 | 是 | 最长50字符，不允许特殊字符 | ... |
| 状态 | 枚举 | 是 | 待处理/进行中/已完成 | ... |

**业务规则**：
1. 规则1：当xxx时，系统应xxx
2. 规则2：xxx不允许重复

**异常场景**：
| 场景 | 系统行为 | 提示信息 |
|---|---|---|
| 名称重复 | 阻止提交 | "名称已存在，请修改" |
| 无权限 | 隐藏入口 | - |

### 2. {功能点名称}
...

## 状态机设计（如适用）
| 当前状态 | 触发事件 | 目标状态 | 约束条件 |
|---|---|---|---|
| 待处理 | 开始处理 | 进行中 | 仅负责人可操作 |
| 进行中 | 完成 | 已完成 | 需填写完成说明 |

## 列表页规格
**筛选条件**：
| 筛选项 | 类型 | 默认值 |
|---|---|---|
| 状态 | 下拉单选 | 全部 |
| 时间范围 | 日期区间 | 最近30天 |

**列表字段**：
| 字段 | 宽度 | 排序 | 操作 |
|---|---|---|---|
| 名称 | 200px | 不支持 | 点击进入详情 |
| 状态 | 100px | 支持 | - |
| 创建时间 | 160px | 默认降序 | - |

**批量操作**：删除、导出

## 接口需求
| 接口 | 方法 | 说明 | 特殊要求 |
|---|---|---|---|
| 列表查询 | GET | 分页+筛选 | 支持模糊搜索 |
| 新增 | POST | 创建记录 | 名称唯一校验 |
| 修改 | PUT | 更新记录 | 乐观锁控制 |
| 删除 | DELETE | 逻辑删除 | 有关联数据时禁止 |
```

### Phase 3: 用户确认

1. 输出后询问是否需要补充或调整
2. 确认后保存到 `docs/requirements/` 目录
3. 文件命名：`[需求文件]{功能模块名}.md`

## 输出规则

- 所有输出使用中文
- 字段规则必须具体到可编码程度（长度、格式、枚举值）
- 异常场景必须覆盖常见边界情况
- 状态机必须明确转换条件和约束
- 权限描述必须到操作级别
