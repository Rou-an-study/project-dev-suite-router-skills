# 需求分析 Skill

将用户的粗略想法转化为结构化的售前需求文档。

## 触发场景

- 用户描述一个新产品/功能的想法
- 用户要求做需求分析或写 PRD
- 用户说"帮我分析一下这个需求"

## 执行流程

### Phase 1: 需求澄清

1. 理解用户的核心诉求，提出 3-5 个关键澄清问题：
   - 目标用户是谁？
   - 核心要解决什么问题？
   - 有哪些已知约束（时间、技术、预算）？
   - 是否有参考产品或竞品？
   - MVP 范围是什么？
2. 等待用户回答后再进入下一阶段。

### Phase 2: 生成并保存需求文档

按以下结构生成 Markdown 文档，并立即保存到项目的 `docs/requirements/` 目录。

保存规则：
- 文件名格式：`[售前需求]{项目名称}.md`
- 如果 `docs/requirements/` 目录不存在，先创建目录
- 保存后在对话中简要说明文件路径，并展示文档摘要或关键章节
- 不要只在对话中展示完整文档而不落盘

```markdown
# [售前需求]{项目名称}

## 文档目的
一句话说明本文档要解决什么问题。

## 需求背景与目标
1. 当前痛点（2-3条）
2. 期望目标（2-3条）

## 范围定义
1. 输入范围
2. 输出范围
3. 不在本期范围

## 核心对象
| 对象 | 业务含义 | 关键信息 |
|---|---|---|
| ... | ... | ... |

## 需求分析与任务拆解
### 1. 业务主任务
### 2. 上下游依赖任务
### 3. 子任务分层（T1-T5）

## 业务功能菜单
（列出一级菜单）

## 菜单具体功能
（每个菜单下的功能点）

## 业务流程走向
### 流程A: {主流程名}
步骤1 -> 步骤2 -> ... -> 步骤N

## 业务接口文档（产品接口清单）
| 接口名称 | 业务用途 | 输入 | 输出 |
|---|---|---|---|

## 数据表结构设计（业务模型）
### 1. {表名}
| 字段 | 说明 |
|---|---|

## 里程碑建议
1. M1: ...
2. M2: ...

## 验收标准
1. ...
2. ...
```

### Phase 3: 用户确认

1. 保存文档后询问用户是否需要调整
2. 根据反馈直接修改已保存的 Markdown 文件
3. 每次修改后说明更新的文件路径和主要变更
4. 用户确认后，将该文件视为 Phase 1 的正式售前需求文档

## 输出规则

- 所有输出使用中文
- 内容简洁但完整，避免空洞描述
- 数据表设计只到业务模型层面，不涉及具体实现细节
- 接口清单只描述业务用途，不涉及技术实现
- 流程描述使用 `->` 箭头连接，一行一个流程


## V1.1 产品定位补强

以下要求优先于前文中较少覆盖产品边界的部分。

生成需求文档前，必须澄清并写入 `docs/PRD.md`：

- 目标用户、使用场景和核心痛点。
- 用户当前的替代方案、竞品和差异化。
- MVP 必须交付的功能，以及明确不做的功能。
- 产品形态：网页、移动端、插件、内部工具、API 或其他。
- 时间、预算、模型/API、部署、数据和合规约束。
- 关键假设、风险和成功指标。

不要把“可能以后需要”的能力放进 MVP。产品范围发生变化时，先更新 PRD 的范围变更记录并得到用户确认，再拆任务或实现。


## V1.2 已有项目的版本需求

当用户要求为已有项目增加功能或发布新版本时，不重新执行完整产品定位。先读取现有 PRD、路线图和相关版本记录，再生成版本范围补充：

- 本版本目标、用户价值和成功指标。
- 本版本功能、非目标和延期项。
- 对既有用户流程、权限、数据、接口、模型成本和合规的影响。
- 是否需要更新 DESIGN 或 ARCHITECTURE。
- 可验证的版本验收标准和回归范围。

将结果记录在 `docs/releases/{版本号}/VERSION-PLAN.md`，必要的产品范围变更同步回写 PRD。

## V1.3.1 PRD 轻量风险标记

在 PRD 的功能清单或核心流程中，为每个功能轻量标记风险类型：资金、权限、数据或普通。只说明风险类型和出错后果，不在需求阶段展开测试用例。

示例字段：

| 功能 | 风险类型 | 出错后果 | 是否发布前重点测试 |
|---|---|---|---|
| 下单支付 | 资金/数据 | 金额错误、库存错误、订单状态错误 | 是 |
| 个人资料展示 | 数据 | 泄露或展示错误用户信息 | 是 |
| 帮助页展示 | 普通 | 信息展示错误 | 否 |

这些标记只作为发布前风险测试门禁的输入。
## V1.4 第一性原理补充

需求阶段对高影响产品决策使用第一性原理：先还原用户目标、业务约束、成功标准和不可破坏规则，再参考竞品或成熟方案。普通低风险功能不做过度推导，避免降低开发效率。
