# 技术架构设计 Skill

根据需求文档生成完整的技术架构设计方案。

## 触发场景

- 用户要求做技术架构设计
- 用户说"帮我设计一下技术方案"
- 需求文档确认后进入技术设计阶段

## 执行流程

### Phase 1: 分析项目特征

读取售前需求文档，提取以下信息用于技术选型判断：

| 判断维度 | 影响选型的因素 |
|---|---|
| 业务复杂度 | 简单CRUD / 复杂业务逻辑 / 高并发 / 实时通信 |
| 团队规模 | 独立开发 / 小团队 / 大团队 |
| 生态需求 | 是否需要丰富的中间件/ORM/权限框架 |
| 性能要求 | 普通Web / 高并发 / 计算密集 / IO密集 |
| 前端需求 | 有前端 / 纯后端API / 移动端 |
| 部署环境 | 云服务 / 私有化 / 嵌入式 / Serverless |
| 项目周期 | 快速原型 / 长期维护 |

### Phase 2: 智能推荐技术栈

**根据项目特征给出推荐，并说明理由：**

推荐逻辑参考：
- **企业级后台管理系统**（复杂业务、权限、审批流）→ 推荐 Java + Spring Boot（生态最完善、框架支持最多）
- **轻量API服务/AI应用/数据处理** → 推荐 Python + FastAPI（开发快、AI生态好）
- **高并发微服务/云原生** → 推荐 Go + Gin（性能好、部署轻量）
- **Windows生态/企业内部系统** → 推荐 C# + ASP.NET Core（与Windows集成好）
- **全栈快速开发/实时应用** → 推荐 Node.js + NestJS（前后端统一语言）
- **系统级/极致性能** → 推荐 Rust + Actix（安全+性能）

**输出格式：**
```
根据你的项目特征分析：
- 业务类型：{xxx}
- 复杂度：{xxx}
- 关键需求：{xxx}

推荐技术栈：{语言} + {框架}
推荐理由：
1. {理由1}
2. {理由2}
3. {理由3}

如果你有其他语言偏好，告诉我即可调整。
```

### Phase 3: 确认技术栈

- 如果用户说"可以"/"就这个" → 使用推荐方案
- 如果用户说"我想用 Go"/"换成 Python" → 尊重用户选择，但如果选择明显不适合，给出风险提示：
  ```
  可以用 {用户选择}，不过需要注意：
  - {潜在问题1，如：该语言在权限框架方面生态较弱，需要自己实现}
  - {潜在问题2}
  
  如果你确认使用，我按这个方案继续。
  ```
- 用户确认后，锁定技术栈进入下一步

### Phase 4: 搜索最佳实践

根据确定的技术栈联网搜索：
- 官方推荐项目结构
- 最新稳定版本号
- 社区最佳实践

### Phase 5: 生成架构文档

```markdown
# [技术架构]{项目名称}

## 1. 系统概述

## 2. 技术选型
| 层次 | 技术 | 版本 | 选型理由 |
|---|---|---|---|
| 后端 | {确定的} | {最新稳定版} | {理由} |
| 前端 | {确定的} | {最新稳定版} | {理由} |
| 数据库 | {确定的} | ... | {理由} |

## 3. 系统架构图（Mermaid）

## 4. 模块划分
（根据所选语言/框架的惯例组织）

## 5. 数据库详细设计
（DDL 语法根据所选数据库调整）

## 6. 接口设计（RESTful）

## 7. 统一规范
（根据所选语言的社区规范）
```


根据架构复杂度选择可视化工具：
- **简单架构**（模块<10个）→ Mermaid 架构图 + HTML 关联图
- **复杂架构**（多层、微服务、云部署）→ 使用 Draw.io MCP 生成专业架构图
- **用户指定** → 遵从用户选择

Draw.io 特别适合需要云服务图标（AWS/GCP/Azure）的架构图。

### Phase 7: 用户确认

## 关键原则

1. **先推荐后确认**：根据项目特征智能推荐，不是直接问"你要用什么"
2. **推荐要有理由**：每个推荐必须说明为什么适合这个项目
3. **尊重用户选择**：用户坚持用其他语言时尊重，但给出风险提示
4. **不盲目推荐**：不要因为自己熟悉某语言就推荐，要基于项目实际需求
5. **版本用最新**：联网确认当前最新稳定版本号
6. **遵循语言惯例**：确定语言后，后续所有设计遵循该语言的社区规范

## 输出规则

- 所有输出使用中文
- 接口设计遵循 RESTful 规范
- 保存到 `docs/architecture/` 目录


## V1.1 架构事实源

架构阶段必须创建或更新 `docs/ARCHITECTURE.md`，它是编码、评审和发布前反扫的技术事实源。除技术栈外，必须记录：

- 模块职责、目录边界、允许依赖与禁止依赖。
- 数据、接口、错误处理、日志、监控和安全约束。
- AI/外部服务的 Provider、超时、重试、降级、成本、限流和数据处理规则。
- 环境变量、部署、回滚和不可破坏规则。
- 可验证的架构验收标准。

有 UI 的项目先读取 `docs/DESIGN.md`；技术方案不得违背 PRD 的 MVP 和成本边界。依赖版本在实际落地时联网确认当前稳定版本。


## V1.2 版本与维护影响分析

版本迭代或维护修复前，架构分析必须列出受影响的模块、数据迁移、接口兼容性、权限、安全、性能、外部服务和部署风险。对破坏性变更提供兼容、迁移或回滚方案。

小修复也必须判断是否触及不可破坏规则；若触及，先扩大评审和测试范围，而不是只按代码行数判断风险。

## V1.3.1 架构安全边界输入

架构文档需要为发布前风险测试提供边界输入，但不在架构阶段执行攻防测试。至少说明：

- 权限边界：角色、租户、管理员能力、跨用户访问限制。
- 数据边界：哪些数据可读、可改、可删、可导出，哪些字段敏感。
- 接口边界：关键接口、内部接口、外部回调、幂等和兼容要求。
- 安全边界：认证、鉴权、字段白名单、限流、防刷、日志脱敏、密钥管理。
- 不可破坏规则：资金、权限、数据相关的硬约束。

这些内容用于发布前安全攻防测试定位重点，不替代发布前实测。
## V1.4 架构推导与对抗式检查

架构选型先从项目约束和不可破坏规则出发，再复用成熟技术方案。对高影响架构变更做轻量对抗式检查：依赖挂了怎么办、权限边界能否绕过、数据是否会不一致、能否回滚、是否影响资金/权限/数据。
