# 测试生成 Skill

根据需求文档和已有代码生成测试用例，并执行验证。

## 触发场景

- 代码生成完成后，进入测试阶段
- 用户说"帮我写测试"、"生成测试用例"
- 用户要求验证某个模块的正确性

## 测试策略（测试金字塔）

```
        /  E2E 测试  \          ← 少量，验证关键用户流程
       / 集成测试     \         ← 适量，验证模块间交互
      / 单元测试       \        ← 大量，验证单个函数/方法
     ──────────────────
```

| 层级 | 数量 | 速度 | 覆盖范围 | 验证什么 |
|---|---|---|---|---|
| 单元测试 | 多（70%） | 快 | 单个函数/方法 | 业务逻辑正确性 |
| 集成测试 | 中（20%） | 中 | 模块间交互 | 接口调用、数据库操作 |
| E2E测试 | 少（10%） | 慢 | 完整用户流程 | 关键业务路径端到端 |

## 执行流程

### Phase 1: 分析测试范围

1. 读取项目的需求文档（`[需求文件]xxx.md`）
2. 读取已有代码结构
3. 识别需要测试的模块和关键业务逻辑
4. 确定测试优先级：
   - P0：核心业务流程（必须测）
   - P1：重要分支逻辑（应该测）
   - P2：边界条件和异常场景（建议测）

### Phase 2: 选择测试框架

根据项目技术栈自动选择：

| 技术栈 | 单元测试 | 集成测试 | E2E测试 |
|---|---|---|---|
| Java + Spring Boot | JUnit 5 + Mockito | Spring Boot Test + TestContainers | - |
| Python + FastAPI | pytest + pytest-mock | pytest + httpx (TestClient) | - |
| Go + Gin | testing + testify | testing + httptest | - |
| C# + ASP.NET Core | xUnit + Moq | WebApplicationFactory | - |
| Node.js + NestJS | Jest + ts-jest | supertest | - |
| Vue 3 前端 | Vitest | Vue Test Utils | Playwright / Cypress |

### Phase 3: 生成单元测试

**针对 Service/业务逻辑层：**

```
测试文件命名：{模块名}ServiceTest.{ext}（Java）/ test_{模块名}.py（Python）/ {模块名}_test.go（Go）

每个业务方法生成：
1. 正常流程测试（happy path）
2. 边界条件测试（空值、极值、临界值）
3. 异常场景测试（参数错误、权限不足、数据不存在）
4. 状态流转测试（如果有状态机）
```

**测试用例结构（通用）：**
```
测试方法命名：test_{被测方法}_{场景}_{预期结果}
示例：test_createLeaveRequest_normalInput_success
      test_createLeaveRequest_dateInPast_throwException
      test_approveRequest_noPermission_forbidden
```

### Phase 4: 生成集成测试

**针对 Controller/API 层：**

```
每个接口生成：
1. 正常请求 → 验证响应状态码和数据结构
2. 参数校验 → 缺少必填字段、格式错误
3. 权限校验 → 未登录、无权限
4. 业务校验 → 数据不存在、状态不允许
```

**配合 MCP**：`mysql-custom` — 准备测试数据、验证数据库状态

### Phase 5: 生成 E2E 测试（可选）

**仅针对关键业务流程：**

```
从需求文档的"业务流程走向"提取关键路径：
例如：创建请假 → 提交审批 → 审批通过 → 状态变更

每个关键路径生成一个 E2E 测试，验证端到端流程完整性。
```

**配合 MCP**：`chrome-devtools` — 前端 E2E 测试时操作浏览器

### Phase 6: 执行测试并报告

1. 运行全部测试
2. 收集测试结果：
   - ✅ 通过数量
   - ❌ 失败数量 + 失败原因
   - ⚠️ 跳过数量
3. 对失败的测试分析原因：
   - 测试用例写错了 → 修复测试
   - 代码有 bug → 报告 bug，进入迭代修复
4. 输出测试报告

**测试报告格式：**
```
## 测试报告

执行时间：{时间}
技术栈：{语言} + {测试框架}

### 结果概览
- 总用例数：{N}
- 通过：{N} ✅
- 失败：{N} ❌
- 跳过：{N} ⚠️
- 覆盖率：{N}%

### 失败用例分析
| 用例 | 失败原因 | 建议修复 |
|---|---|---|
| test_xxx | 预期200实际返回403 | 检查权限配置 |

### 下一步
- [ ] 修复失败用例对应的代码bug
- [ ] 补充遗漏的边界测试
```

## 测试文件目录规范

```
# Java
src/test/java/com/{group}/{artifact}/
├── service/
│   └── {Module}ServiceTest.java      # 单元测试
├── controller/
│   └── {Module}ControllerTest.java   # 集成测试
└── e2e/
    └── {Flow}E2ETest.java            # E2E测试

# Python
tests/
├── unit/
│   └── test_{module}_service.py
├── integration/
│   └── test_{module}_api.py
└── e2e/
    └── test_{flow}.py

# Go
{module}/
├── service_test.go                   # 单元测试
├── handler_test.go                   # 集成测试
└── e2e_test.go                       # E2E测试

# Vue 前端
tests/
├── unit/
│   └── {Component}.spec.ts
├── integration/
│   └── {Page}.spec.ts
└── e2e/
    └── {flow}.spec.ts
```

## 与迭代开发的关系

```
代码生成（Phase 7）
    ↓
测试生成 + 执行（Phase 8）← 本 skill
    ↓
    ├── 全部通过 → 进入下一模块
    └── 有失败 → 分析原因 → 修复代码 → 重新测试（循环）
```

## 输出规则

- 测试代码注释使用中文
- 测试方法命名使用英文（遵循各语言测试命名规范）
- 测试数据使用中文模拟数据（贴近真实业务）
- 每个测试方法必须有清晰的 Given-When-Then 结构
- 测试文件保存到项目标准测试目录


## V1.2 回归测试规则

新版本和维护修复不能只验证新增代码。测试计划必须覆盖：

- 新功能或修复问题的目标验收。
- 受影响的既有主流程。
- 受影响的数据迁移、接口兼容性、权限与错误路径。
- 有 UI 时的加载、空、错误和关键交互状态。
- 有 AI/外部服务时的超时、失败、限流或降级路径。

将回归范围和结果记录在版本计划或维护记录中。无法执行的测试必须说明原因与风险，不得默认视为通过。

## V1.3 风险驱动功能测试门禁

本规则在本次版本范围内的功能开发完成后、版本发布前执行；单个 TODO 完成后不重复执行完整门禁。不要追求一开始全量覆盖，先锁定资金、权限、数据相关高风险功能。

### 高风险功能识别

从 PRD、TODO、VERSION-PLAN、MAINTENANCE-RECORD、接口文档和代码中识别：

- 会影响资金的功能：支付、退款、金额、优惠、结算、余额、库存扣减等。
- 会影响权限的功能：登录、角色、管理员接口、授权、租户隔离、审批等。
- 会影响数据的功能：查看、创建、修改、删除、导出、导入、跨用户数据访问等。

输出到 `docs/tests/RISK-BASED-TEST-PLAN.md`：必须重点测试清单、暂不重点测试清单、每个功能出错后果和风险分类。

### 三类用例

只对高风险清单上的功能设计深度用例：

1. 正常且结果正确：功能该成功时必须成功，且计算结果、状态变化、数据变化正确。
2. 边界临界：0、负数、超大值、最后一个库存、并发临界、时间边界、权限边界等。
3. 异常与非法输入：未登录、越权、伪造身份、非法字符、注入、字段篡改、绕过前端限制等。

判断标准不是“接口返回 200”，而是业务结果正确。例如金额是否正确扣减、库存是否正确变化、订单状态是否符合预期、越权请求是否被拒绝且没有产生副作用。

### 自动化命令

将确认通过的关键测试沉淀到 `docs/tests/TEST-AUTOMATION-RUNBOOK.md`：

- 测试命令。
- 运行前置条件。
- 如何判断通过或失败。
- 本次运行结果。
- 单个任务完成后需要局部复跑的范围。
- 版本发布前需要完整复跑的范围。

### 测试真实性验证

生成 `docs/tests/TEST-EFFECTIVENESS-REPORT.md`，逐个说明每个测试真正验证了什么业务结果、是否真实调用被测功能、是否检查数据库或关键状态变化。

对关键功能测试做受控故障注入：故意把金额、权限、状态或数据变化逻辑改错，运行对应测试。测试必须失败；如果仍然通过，说明是假测试，必须重写。故障注入必须可回滚，验证后恢复代码，不得污染正式变更。
