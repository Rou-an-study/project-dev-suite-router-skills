# 需求转化 Skill

将售前需求文档转化为可执行的产品需求文件。

## 触发场景

- 用户说"把售前需求转成产品需求"
- 售前需求确认后需要进入开发阶段
- 用户要求生成 `[需求文件]` 格式的文档

## 转化规则

### 文件命名转换
| 源文件 | 目标文件 |
|---|---|
| `[售前需求]xxx.md` | `[需求文件]xxx.md` |
| `售前原型-xxx.html` | `需求原型-xxx.html` |
| `[业务流程图]xxx.mmd` | 保持不变（直接复用） |
| `[业务关联图]xxx.html` | 保持不变（直接复用） |

### 内容转化策略

**方式A：直接复制（快速）**
- 适用于售前需求已经足够详细的情况
- 仅复制文件并重命名，不修改内容
- 原型文件同理

**方式B：细化补充（推荐）**
- 在售前需求基础上补充以下内容：
  - 字段级别的规则定义
  - 接口的请求/响应详细结构
  - 数据库字段的类型和约束
  - 异常场景和边界条件
  - 权限矩阵细化

## 执行流程

### Phase 1: 识别源文件

1. 扫描项目 `docs/requirements/` 目录
2. 找到所有 `[售前需求]` 开头的文件
3. 列出待转化的文件清单

### Phase 2: 确认转化方式

询问用户：
- 直接复制（仅重命名）？
- 还是需要细化补充？

### Phase 3: 执行转化

**如果直接复制：**
1. 复制 `[售前需求]xxx.md` → `[需求文件]xxx.md`
2. 复制 `售前原型-xxx.html` → `需求原型-xxx.html`（如果存在）

**如果细化补充：**
1. 读取售前需求文档
2. 调用 `detail-requirements` skill 的逻辑进行细化
3. 输出为 `[需求文件]xxx.md`

### Phase 4: 输出

- 保存到 `docs/requirements/` 目录
- 原型文件保存到 `docs/prototypes/` 目录

## 输出规则

- 转化后的文件必须保留售前需求的所有业务信息
- 不允许删减业务内容，只允许补充细化
- 文件编码统一 UTF-8
