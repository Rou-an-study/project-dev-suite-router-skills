# 项目开发流程 Skills 学习文档

> 当前版本：`project-dev-suite-router v1.5.0`  
> 适用对象：希望用 AI 辅助完成软件项目开发、版本迭代、上线测试、维护修复的人  
> 核心目标：让 AI 不只是“写代码”，而是按照可验证、可回退、可迭代的项目工程流程协作。

---

## 1. 这套 Skills 是什么

这套 `project-dev-suite-router` 不是一个默认参与所有任务的普通开发 Skill，而是一套**隔离的项目开发流程套件**。

它的作用不是让 AI 一上来就写代码，而是帮助我们在合适的时候，按照项目工程流程完成：

- 产品定位
- PRD 梳理
- 设计契约
- 架构设计
- 任务拆解
- 代码开发
- 测试验证
- 安全攻防
- 发布检查
- 版本迭代
- 维护热修复

它最重要的原则是：

```text
先判断任务有多重，再决定流程走多深、上下文读多少、测试做多深。
```

这意味着它不是“每次都完整跑一遍流程”，而是根据任务大小选择不同强度。

---

## 2. 为什么需要这套流程

AI 写代码很快，但如果没有流程约束，很容易出现这些问题：

- 需求没有澄清，代码是在猜。
- MVP 范围不清，越写越大。
- 架构没有边界，文件和模块越改越乱。
- 测试只看接口成功，不看业务结果。
- 权限、数据、资金、安全问题被忽略。
- 每次新会话都丢失上下文。
- 发布前不知道到底有没有准备好。
- 修 bug 时顺手做重构，引入新问题。

这套 Skills 的价值就是给 AI 加上项目护栏：

```text
让 AI 能理解、能执行、能验证、能回滚、能继续迭代。
```

---

## 3. 什么时候应该启用

### 3.1 应该启用的场景

适合启用项目开发套件的情况：

- 从 0 到 1 做一个软件项目。
- 规划一个明确版本，例如 V1.1、V2.0。
- 给已有项目新增一组有业务意义的功能。
- 准备发布上线。
- 做高风险 hotfix。
- 处理安全、权限、数据、资金相关变更。
- 需要完整项目上下文和工程流程治理。

### 3.2 不应该启用的场景

不适合启用完整项目套件的情况：

- 普通问答。
- 代码解释。
- 单文件小修。
- 临时脚本。
- 文档、图片、表格等非软件项目任务。
- 很小的 bug 修复。
- 不需要项目级上下文的任务。

这类任务应该保持普通 Codex 行为，不加载项目开发套件。

## 4. V1.5 的核心改进：流程强度分级

`v1.5.0` 最重要的变化是增加了 `L0 / L1 / L2 / L3` 四档流程强度。

| 强度 | 名称 | 适用场景 | 是否启用项目套件 |
|---|---|---|---|
| L0 | 不启用项目套件 | 普通问答、代码解释、单文件小修、小脚本 | 否 |
| L1 | 轻量上下文模式 | 已有项目中的小功能、小 bug、小优化 | 是，但只读最小上下文 |
| L2 | 标准项目迭代模式 | 一个明确版本或一组功能 | 是，读取项目事实源并规划迭代 |
| L3 | 发布或高风险门禁模式 | 发布前、重大合并、高风险 hotfix、资金/权限/数据/安全变更 | 是，执行完整风险测试与安全门禁 |

### 4.1 L0：不启用项目套件

适合：

- 解释代码。
- 修改一个小函数。
- 回答技术问题。
- 临时生成脚本。

原则：

```text
不要为了小任务启动大流程。
```

### 4.2 L1：轻量上下文模式

适合：

- 已有项目中修一个小 bug。
- 改一个局部功能。
- 优化一个小模块。

读取内容：

- 当前相关代码。
- 当前 TODO 或相关任务说明。
- 必要的架构片段。
- 如果涉及 UI，才读 DESIGN。
- 如果涉及产品边界，才读 PRD。

验证方式：

- 局部测试。
- lint / typecheck。
- 必要的局部风险验证。

### 4.3 L2：标准项目迭代模式

适合：

- 一个明确版本。
- 一组相关功能。
- 已有项目的能力升级。

读取内容：

- `PRD.md`
- `ARCHITECTURE.md`
- `TODO.md`
- `VERSION-PLAN.md`
- 相关代码
- 涉及 UI 时读取 `DESIGN.md`

验证方式：

- 范围内单元测试。
- 接口测试。
- 集成测试。
- 受影响路径回归。

### 4.4 L3：发布或高风险门禁模式

适合：

- 版本发布前。
- 重大功能集合合并前。
- 高风险 hotfix 发布前。
- 涉及资金、权限、核心数据、安全边界的变更。

执行内容：

- 风险驱动功能测试。
- 测试自动化运行手册。
- 测试真实性验证。
- 安全攻防测试。
- 问题清单与方案核对。
- 修复后回归。

## 5. 项目状态识别

启用项目套件后，不应该直接开始写代码，而应该先做一次项目状态识别。

推荐输出格式：

```text
当前识别：
- 路径：新项目 / 版本迭代 / 维护热修复
- 强度：L1 / L2 / L3
- 当前阶段：需求 / 设计 / 架构 / 开发 / 测试 / 发布前 / 维护
- 已有事实源：PRD / DESIGN / ARCHITECTURE / TODO / VERSION-PLAN / MAINTENANCE-RECORD / 测试报告
- 缺失或冲突事实：
- 高风险区域：资金 / 权限 / 数据 / 安全 / AI 成本隐私 / 无
- 下一步：
```

这一步的意义是：

- 避免把已有项目当新项目重新规划。
- 避免把小任务误判成完整项目流程。
- 避免读取过多上下文。
- 提前发现文档缺失或冲突。

## 6. 上下文加载预算

这套 Skills 的一个重要目标是减少 token 浪费。

默认规则：

```text
先读取完成当前任务所需的最小上下文。
只有发现冲突、缺口或高风险时，再扩大读取范围。
```

### 6.1 L1 上下文预算

只读：

- 当前任务。
- 相关代码。
- 必要的 `ARCHITECTURE.md` 或 `TODO.md` 片段。

按需读取：

- UI 变更才读 `DESIGN.md`。
- 产品范围不清才读 `PRD.md`。

### 6.2 L2 上下文预算

读取：

- `PRD.md`
- `ARCHITECTURE.md`
- `TODO.md`
- `VERSION-PLAN.md`
- 相关代码

按需读取：

- 涉及 UI 读 `DESIGN.md`。
- 涉及数据库读数据库设计。
- 涉及接口读 API 文档。
- 涉及权限读安全边界说明。

### 6.3 L3 上下文预算

在 L2 基础上读取：

- `RELEASE-READINESS.md`
- `RISK-BASED-TEST-PLAN.md`
- `TEST-AUTOMATION-RUNBOOK.md`
- `TEST-EFFECTIVENESS-REPORT.md`
- `SECURITY-ATTACK-REPORT.md`

## 7. 完整项目开发流程

下面是完整项目流程。实际执行时，不一定每次都从头走到尾，而是根据项目状态和强度选择对应阶段。

---

## 8. 第 0 步：触发判断

### 要做什么

- 判断用户请求是不是项目级任务。
- 判断是否需要启用项目开发套件。
- 判断初始强度是 L0、L1、L2 还是 L3。
- 在需要启用套件时，先征求用户确认。

### 可验证点

- 是否经过用户确认。
- 是否选择了正确强度。
- 是否避免普通任务误触发完整流程。

### 循环迭代

如果一开始判断为 L1，但过程中发现涉及权限、核心数据或发布风险，可以升级到 L2 或 L3。  
如果发现只是普通小改动，也可以降级为 L0 或 L1。

---

## 9. 第 1 步：产品定位与 PRD

### 适用场景

- 新项目。
- 大版本方向调整。
- 产品范围不清楚的功能迭代。

### 要做什么

- 明确产品解决什么问题。
- 明确目标用户是谁。
- 明确用户现在怎么解决。
- 明确现有方案有什么不足。
- 明确竞品或替代方案。
- 明确 MVP 先做什么。
- 明确哪些功能暂时不做。
- 明确产品形态，例如网页、App、小程序、插件、内部工具。
- 明确成本约束，例如 AI API 成本、服务器成本、时间成本。
- 标记高风险功能，尤其是资金、权限、数据。

### 核心产物

```text
docs/PRD.md
```

### 可验证点

- 是否有明确目标用户。
- 是否有明确 MVP。
- 是否有明确非目标。
- 是否有验收标准。
- 是否标记资金、权限、数据风险。
- 是否经过用户确认。

### 循环迭代

```text
后续设计 / 架构 / 开发发现 PRD 不合理 -> 回到 PRD 修正
```

PRD 可以演进，但不能悄悄修改，必须有变更依据。

---

## 10. 第 2 步：设计契约与原型

### 适用场景

- 有 UI 的项目。
- 需要明确页面、交互、视觉风格的项目。

### 要做什么

- 形成 `DESIGN.md`。
- 明确整体视觉气质。
- 明确布局规则。
- 明确组件风格。
- 明确交互状态。
- 明确加载、空状态、错误状态。
- 生成 UI 原型。
- 用原型反向检查 PRD 是否逻辑顺畅。

### 核心产物

```text
docs/DESIGN.md
原型文件
流程图
页面关系说明
```

### 可验证点

- 核心用户路径是否走得通。
- 页面是否覆盖 MVP。
- 交互状态是否完整。
- 原型是否暴露产品逻辑问题。
- 设计是否和 PRD 一致。

### 循环迭代

```text
PRD -> DESIGN -> 原型 -> 发现问题 -> 回 PRD 或 DESIGN 修正
```

---

## 11. 第 3 步：架构设计

### 适用场景

- 新项目。
- 版本迭代涉及技术结构变化。
- 权限、数据、接口、AI 服务、部署方式有变化。

### 要做什么

- 选择技术栈。
- 明确前端、后端、数据库、AI 服务、外部服务。
- 明确模块边界。
- 明确目录结构。
- 明确数据流。
- 明确权限边界。
- 明确 API 边界。
- 明确安全边界。
- 明确不可破坏规则。
- 明确部署方式和成本风险。
- 对高影响决策使用轻量第一性原理。

### 第一性原理的使用方式

第一性原理不是拒绝成熟方案，也不是从零发明一切。

它只用于高影响决策：

- 产品定位。
- MVP 边界。
- 技术选型。
- 权限模型。
- 资金规则。
- 数据一致性。
- AI 成本与隐私。
- 多租户和安全边界。

执行方式：

```text
先拆目标、约束、不可破坏规则 -> 再参考成熟方案 -> 最后判断哪些可复用、哪些不能照搬
```

### 核心产物

```text
docs/ARCHITECTURE.md
```

### 可验证点

- 技术栈是否能实现 PRD。
- 架构是否能指导代码。
- 权限、数据、接口边界是否清楚。
- 成本是否可接受。
- 是否避免过度设计。
- 是否经过用户确认。

### 循环迭代

```text
PRD -> 架构 -> 发现成本 / 技术 / 安全问题 -> 回 PRD 或架构调整
```

---

## 12. 第 4 步：两段式任务规划

### 要做什么

第一段：路线图。

- 明确里程碑。
- 明确范围。
- 明确依赖。
- 明确风险。

第二段：TODO。

- 拆成可执行任务。
- 每个任务明确目标。
- 明确允许修改范围。
- 明确不可破坏项。
- 明确验收标准。
- 明确测试要求。
- 明确依赖和状态。

### 核心产物

```text
docs/TODO.md
必要时：ROADMAP.md
```

### 单个 TODO 完成定义

- 功能或修复已完成。
- 没有突破允许修改范围。
- 没有破坏不可破坏项。
- 相关测试通过。
- 涉及资金、权限、数据时，完成局部风险验证。
- TODO 状态更新。
- 必要时完成 Git 提交。

### 可验证点

- TODO 是否足够小。
- 每个 TODO 是否能独立执行。
- 是否有明确验收标准。
- 是否标记风险。
- 是否写清楚不能改什么。

### 循环迭代

```text
任务过大 -> 继续拆分
需求变化 -> 回 PRD / VERSION-PLAN
架构限制变化 -> 回 ARCHITECTURE
```

---

## 13. 第 5 步：详细需求、接口与数据库设计

### 要做什么

- 按当前里程碑细化需求。
- 设计接口。
- 设计数据库。
- 明确请求、响应、错误码。
- 明确字段、索引、约束。
- 明确权限校验。
- 明确数据状态变化。
- 明确异常路径。

### 核心产物

```text
详细需求文档
API 文档
数据库 SQL
数据模型说明
```

### 可验证点

- 接口是否覆盖当前 TODO。
- 数据库是否支撑业务状态。
- 后端是否承担关键校验。
- 权限、数据边界是否和架构一致。
- 是否避免只靠前端限制。

### 循环迭代

```text
接口设计发现数据模型不够 -> 回数据库设计
数据库发现业务状态不清 -> 回详细需求
权限边界不清 -> 回架构
```

---

## 14. 第 6 步：代码骨架与冒烟测试

### 要做什么

- 根据架构生成或整理项目骨架。
- 建立基础目录、依赖、启动配置。
- 接好数据库或外部服务配置。
- 做最小可运行链路。
- 不急着写完整业务，先确认项目能跑。

### 冒烟测试检查

- 项目能安装依赖。
- 项目能编译。
- 服务能启动。
- 健康检查接口可访问。
- 数据库连接正常。
- 至少一个基础接口能跑通。
- 如果有登录，认证链路能走通。
- 如果有前端，页面能打开。

### 可验证点

不是“代码看起来对”，而是服务真的跑起来。

冒烟失败不能进入正式迭代。

### 循环迭代

```text
骨架生成 -> 冒烟测试 -> 失败 -> 修配置 / 依赖 / 启动问题 -> 再冒烟
```

---

## 15. 第 7 步：小步迭代开发

这是日常开发的主循环。

### 要做什么

- 从 TODO 中取一个任务。
- 读取当前任务需要的最小上下文。
- 明确允许修改范围。
- 明确不可破坏项。
- 实现代码。
- 执行相关测试。
- 通过后更新 TODO。
- 必要时提交 Git。

### V1.5 对日常开发的关键改进

- 单个普通 TODO 不跑完整 L3 门禁。
- 单个任务只做局部验证。
- 只有涉及资金、权限、核心数据、安全边界时，才做局部风险验证。
- 只有发布前、重大合并前、高风险 hotfix 前，才做完整 L3。

### 可验证点

- 每次只做一个 TODO。
- 代码变化范围可控。
- 测试命令可复跑。
- 结果通过或失败清楚。
- TODO 状态被更新。

### 循环迭代

```text
取 TODO -> 实现 -> 测试 -> 失败则修复 -> 通过 -> 更新状态 -> 下一个 TODO
```

---

## 16. 第 8 步：常规测试

### 要做什么

- 写或补测试。
- 跑单元测试。
- 跑接口测试。
- 跑必要的集成测试。
- 前端项目可跑页面或 E2E 验证。
- 数据库相关功能要验证真实数据变化。
- 不只看接口是否返回成功。

### 测试重点

- 正常情况结果正确。
- 边界情况处理正确。
- 异常输入能被拦截。
- 数据变化符合预期。
- 权限校验有效。

### 可验证点

- 测试是否真的调用被测功能。
- 是否验证业务结果。
- 是否能一条命令或少数命令复跑。
- 失败时是否能定位问题。

### 循环迭代

```text
测试失败 -> 修复 -> 重跑
测试不可信 -> 重写测试
```

---

## 17. 第 9 步：发布前风险驱动测试与安全攻防门禁

这是 L3，不是每个 TODO 都执行。

### 触发场景

- 版本发布前。
- 重大功能集合合并前。
- 高风险 hotfix 发布前。
- 涉及资金、权限、核心数据、安全边界的高风险变更。

### 要做什么

- 识别高风险功能。
- 按资金、权限、数据分类。
- 列出必须重点测的功能。
- 列出暂不重点测的低风险功能。
- 为关键功能设计正常、边界、异常测试。
- 固化自动化测试命令。
- 验证测试真实性。
- 做安全攻防测试。
- 先列问题清单，不直接改。
- 和用户逐条核对后再修。
- 修完后做回归。

### 核心产物

```text
docs/tests/RISK-BASED-TEST-PLAN.md
docs/tests/TEST-AUTOMATION-RUNBOOK.md
docs/tests/TEST-EFFECTIVENESS-REPORT.md
docs/tests/SECURITY-ATTACK-REPORT.md
```

### 安全测试包括

- 数据越权：修改用户 ID、订单 ID，看能否访问或修改别人的数据。
- 功能越权：普通用户调用管理员接口。
- 身份认证绕过：不登录、伪造 token、篡改凭证。
- 偷改字段提权：偷偷传 `role=admin`、`price=0`、`amount=-100`。
- 恶意输入与注入：SQL 注入、命令注入、脚本注入。
- 绕过前端限制：直接调用接口，看后端是否校验。
- 敏感信息泄露：检查是否返回密码、密钥、手机号等。
- 资源滥用与防刷：高频、超大、并发请求，看是否有限流。

### 可验证点

- 有高风险功能清单。
- 有测试命令。
- 有测试真实性验证。
- 有攻击复现证据。
- 有风险等级。
- 有问题后果。
- 有修复方案。
- 有用户核对记录。
- 有修复后回归结果。

### 循环迭代

```text
风险测试 -> 发现问题 -> 列清单和证据 -> 用户核对 -> 修复 -> 回归 -> 再确认
```

## 18. 第 10 步：发布就绪检查

### 要做什么

使用 `RELEASE-READINESS.md` 检查：

- 最终目录、模块、技术栈是否回写架构。
- 环境变量、启动、测试、部署、回滚说明是否完整。
- README 是否更新。
- PRD 验收标准是否满足。
- DESIGN 是否和最终 UI 一致。
- 测试和安全审查记录是否完整。
- 版本计划或维护记录是否关联本次发布。
- Git tag 是否准备或完成。

### V1.5 增加检查

- 本次强度是 L1、L2 还是 L3。
- 有没有对普通 TODO 误执行完整 L3。
- 版本或 hotfix 的影响范围是否确认。
- 单个 TODO、版本、hotfix 的完成定义是否满足。
- 如果没执行 L3，原因是否合理。

### 可验证点

- 发布材料完整。
- 文档和真实代码一致。
- 环境变量不泄露真实密钥。
- 测试结果有记录。
- Git tag 可追踪。

### 循环迭代

```text
发布检查发现缺口 -> 回对应阶段修复 -> 再检查
```

## 19. 版本迭代路径

### 适用场景

- 已有项目新增一组功能。
- 做 V1.1、V1.2、V2.0。
- 不需要从 0 重新做完整产品发现。

### 要做什么

- 读取已有事实源。
- 生成或更新 `VERSION-PLAN.md`。
- 明确本版本目标。
- 明确本版本范围。
- 明确不做什么。
- 明确影响模块。
- 明确数据、API、权限、UI、部署影响。
- 生成本版本 TODO。
- 小步开发。
- 做范围内测试和回归。
- 发布前按需进入 L3。

### 核心产物

```text
docs/releases/{版本号}/VERSION-PLAN.md
```

### 可验证点

- 版本目标清楚。
- 非目标清楚。
- 影响范围清楚。
- 回归范围清楚。
- 完成定义清楚。

### 循环迭代

```text
版本计划 -> TODO -> 开发 -> 测试 -> 发布 -> 反馈 -> 下一版本
```

## 20. 维护 / Hotfix 路径

### 适用场景

- bug 修复。
- 安全问题。
- 性能退化。
- 线上故障。
- 依赖风险。

### 要做什么

- 记录问题。
- 确认严重等级。
- 确认影响用户和版本。
- 明确复现条件。
- 明确最小修复范围。
- 禁止混入无关重构或新功能。
- 做定向测试。
- 做受影响主流程回归。
- 必要时进入精简但严格的 L3。
- 更新维护记录。
- 必要时打 hotfix tag。

### 核心产物

```text
docs/maintenance/MAINTENANCE-RECORD.md
```

### 可验证点

- 问题能复现。
- 修复范围最小。
- 没有混入无关改动。
- 定向测试通过。
- 受影响路径回归通过。
- 维护记录完整。

### 循环迭代

```text
复现 -> 最小修复 -> 定向测试 -> 回归 -> 发布 -> 根因沉淀
```

## 24. 整体评价

当前 v1.5 已经从“完整项目开发流程”升级成了“带调度能力的项目开发操作系统”。

它最重要的变化是：

```text
不再只有一条重流程，而是能根据任务大小选择轻重。
```

### 优点

- 适合从 0 到 1 做项目。
- 适合后续版本迭代。
- 适合 hotfix 和安全修复。
- 有 PRD、DESIGN、ARCHITECTURE、TODO 作为事实源。
- 有发布前风险测试和安全攻防门禁。
- 有测试真实性验证，能防止假测试。
- 有 v1.5 强度分级，能降低日常开发负担。
- 有上下文预算，能减少 token 浪费。
- 有完成定义，能减少“到底做完没有”的歧义。

### 缺点

- 对极小 demo 仍然偏重。
- 如果使用者不按 L0-L3 分级，仍可能过度流程化。
- 文档越多，维护责任越大。
- 如果 PRD、ARCHITECTURE、TODO 过期，会反过来误导开发。
- L3 安全门禁很有价值，但成本也高，不适合频繁执行。

## 26. 用于日常开发会不会加重流程

答案取决于是否正确使用 L0-L3。

### 正确使用时

不会明显加重。

推荐方式：

```text
普通小问题：L0
已有项目小改动：L1
一个版本或一组功能：L2
发布前 / 高风险变更：L3
```

这样用时，它会增加少量前置判断和文档维护成本，但能减少更大的返工成本。

### 错误使用时

会明显加重。

错误方式包括：

- 每个小功能都跑完整 PRD、架构、测试门禁。
- 每个 TODO 都完整读取全部文档。
- 每次都生成完整测试报告。
- 把 L3 当成日常开发默认模式。

## 27. 最佳实践

### 27.1 先判断强度

不要一上来就完整启动项目流程。

先问：

```text
这件事是 L0、L1、L2，还是 L3？
```

### 27.2 先读最小上下文

默认只读完成当前任务需要的内容。

如果发现问题，再扩大读取范围。

### 27.3 文档必须服务开发

PRD、DESIGN、ARCHITECTURE、TODO 都不是形式主义。

它们必须能帮助 AI：

- 理解上下文。
- 控制修改范围。
- 判断风险。
- 验证结果。

### 27.4 高风险功能必须深测

涉及以下内容时，不要只做表面测试：

- 资金。
- 权限。
- 核心数据。
- 用户隐私。
- 安全边界。
- AI 成本与隐私。

### 27.5 发布前必须反扫文档

发布前要确认：

- 文档是否和真实代码一致。
- README 是否能指导下一次接手。
- 环境变量和部署说明是否完整。
- 测试和安全报告是否可追踪。

### 28.5 v1.3：发布前风险驱动测试与安全攻防门禁

```text
docs/tests/TEST-AUTOMATION-RUNBOOK.md

```

新增测试思想：

- 先识别真正高风险功能。
- 重点测试资金、权限、数据。
- 次要功能够用就行，不追求无意义的全面覆盖。
- 测试要验证业务结果，不只是接口返回成功。
- 测试要能自动化复跑。
- 测试要验证真实性，避免假测试。

安全攻防覆盖：

- 数据越权。

```text

```

这个版本解决的问题：

- 只写功能、不验证业务正确性。
- 测试只看接口 200。
- 没有安全攻防视角。
- 假测试无法暴露问题。
- 发布前没有风险证据和修复闭环。

### 28.6 v1.3.1：上游风险标记闭环

v1.3.1 的核心是让上游文档提前留下轻量风险标记。

更新内容：

它没有做的事：

- 没有把完整测试用例塞进 PRD。
- 没有把安全攻防执行塞进 ARCHITECTURE。
- 没有让每个 TODO 都写大量测试脚本。

v1.3.1 的关键思想：

```text
PRD / 架构 / TODO 提供风险输入。
发布前测试门禁集中执行完整测试协议。
```

这个版本解决的问题：

- 发布前才突然识别风险，容易遗漏。
- 上游没有风险标记，测试阶段缺少依据。
- 又不能把完整测试流程平均拆散到所有文档里。

### 28.7 v1.4：轻量推理原则

Git 信息：

```text
Tag: project-dev-suite-v1.4.0
Commit: 8aec793
提交说明：项目开发流程 skills 版本 v1.4
```

这个版本加入了两个思考原则：

```text
第一性原理
对抗式审查
```

但它们被明确限制为“轻量原则”，不是新增流程阶段。

第一性原理的使用边界：

- 产品定位。
- MVP 边界。
- 技术选型。
- 权限模型。
- 资金规则。
- 数据一致性。
- AI 成本和隐私。
- 多租户和安全边界。

使用方式：

```text
先拆目标、约束、不可破坏规则，再参考成熟方案。
```

对抗式审查的使用边界：

- PRD 确认。
- 架构确认。
- TODO 执行前。
- 发布前。
- hotfix 前后。

它要追问：

- 方案会在哪里失败？
- 会怎样被误用或滥用？
- 会不会破坏资金、权限、数据、可靠性、发布安全？

v1.4 的关键思想：

```text
第一性原理和对抗式审查是关键节点的思考姿态，不是新流程、不是新文档负担。
```

这个版本解决的问题：

- 高影响决策不能只照抄成熟方案。
- 但也不能所有小功能都从零推导。
- 安全和风险审查不能只放在最后，也不能让流程变臃肿。

### 28.8 v1.4.1：测试门禁触发粒度修正

Git 信息：

```text
Tag: project-dev-suite-v1.4.1
Commit: f50e15f
提交说明：项目开发流程 skills 版本 v1.4.1
```

这个版本是一次关键修正。

修正的问题：

v1.3 / v1.4 中“功能开发完成后执行风险测试门禁”的表述，容易被误解成：

```text
每完成一个单独功能或 TODO，都要跑完整发布前风险测试与安全攻防门禁。
```

这是不合理的，因为完整门禁成本很高，不应该用于每个小任务。

v1.4.1 明确了触发粒度：

- 单个 TODO 完成后：只执行相关单元测试、接口测试、lint/typecheck，以及必要局部风险验证。
- 单个高风险功能完成后：如果涉及资金、权限或数据，可以提前执行局部风险测试和必要安全验证。
- 版本发布或重大功能集合合并前：执行完整 V1.3 风险测试与安全攻防门禁。
- 高风险 hotfix 发布前：按受影响范围执行精简但严格的风险门禁、定向测试和回归。

v1.4.1 的关键思想：

```text
完整门禁服务发布级风险，不服务每个普通 TODO。
```

这个版本解决的问题：

- 防止流程被误用得过重。
- 防止每个小任务都消耗大量测试和 token。
- 保留高风险功能的局部深测能力。

### 28.9 v1.5：流程强度分级与上下文节省

Git 信息：

```text
Tag: project-dev-suite-v1.5.0
Commit: e409139
提交说明：项目开发流程 skills 版本 v1.5
```

这是当前最新版本。

v1.5 的核心目标不是继续增加流程，而是让流程更会调度。

新增四档强度：

```text
L0：不启用项目套件
L1：轻量上下文模式
L2：标准项目迭代模式
L3：发布或高风险门禁模式
```

新增能力：

- 流程强度分级。
- 上下文加载预算。
- 项目状态识别输出。
- 版本 / hotfix 影响范围确认。
- 单个 TODO、版本发布、hotfix 的完成定义。

v1.5 解决的问题：

- 日常开发不应该总是启用完整流程。
- 不同任务应该读取不同粒度的上下文。
- 进入项目套件后，应该先说明路径、强度、阶段、事实源和下一步。
- 版本迭代和 hotfix 需要更明确影响范围。
- “做完了没有”需要完成定义。

v1.5 的关键思想：

```text
该轻的时候轻，该重的时候重。
```

推荐使用方式：

```text
普通小问题：L0
已有项目小改动：L1
一个版本或一组功能：L2
发布前 / 高风险变更：L3
```

这个版本的价值：

- 明显降低日常开发流程负担。
- 降低 token 消耗。
- 避免误触发完整项目流程。
- 保留发布级和高风险变更的强验证能力。

## 29. 版本演进总结

这套 Skills 的演进路径可以总结为：

```text
v1.0：先把流程隔离出来，能被 Codex 使用
v1.1：建立项目事实源，防止 AI 乱猜上下文
v1.2：支持完整生命周期，不止 MVP
v1.3：加入发布前风险测试和安全攻防门禁
v1.3.1：让上游文档给测试门禁提供风险输入
v1.4：加入轻量第一性原理和对抗式审查
v1.4.1：修正完整门禁触发粒度，避免过重
v1.5：增加流程强度分级和上下文预算，适配日常开发
```

它的方向不是不断加重，而是逐步变得更成熟：

```text
从“完整流程”演进为“可调度、可验证、可迭代、可控制成本的项目开发系统”。
```
